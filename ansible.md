# APIがないサーバーをどう自動化する？SSH地獄からAnsibleに逃げた話

## 3行まとめ

- APIがない端末でも、SSHさえあれば自動化は可能  
- Ansibleなら対象側の開発不要で、安全に構成変更できる  
- 複数台運用・ログ管理・再実行性を考えるなら非常に現実的な選択肢  




---
## 外部連携という課題

外部端末を操作したい場合、第一候補はAPI連携だろう。

APIが提供されていれば理想的だ。  
HTTPで呼び出すだけでよく、認証やレスポンス管理も設計しやすい。いわば「勝ち確」パターンである。

しかし、もし外部端末にAPIが存在しなかったらどうするか？  

ssh + scpで頑張る？、APIを作る
それ、将来地獄になりそうだし、気が重い...
それを解決した話になります。

---

## 想定ケース

例えば：

- ラズパイの設定ファイルを書き換えたい
- 外部サーバーの設定を一括変更したい
- PCの初期セットアップを自動化したい

設定変更用のAPIを実装することもできる。  
だがそれは「対象側にも開発が必要」になる。

- API実装
- 認証設計
- ログ実装
- エラーハンドリング

対象が単なるLinuxサーバーやラズパイである場合、そこまでの開発コストをかけるのは重い。

では、

- SSHで接続して
- scpでファイルを送り
- コマンドを叩く

これでよいのでは？

確かに可能だ。

しかし実際にやろうとすると、

- 本当に送信できたのか？
- 差分はあったのか？
- 失敗時のログは？
- 複数台にどう展開する？
- 途中で止まったらどうなる？

といった「運用設計」が一気に重くなる。

ここで登場するのが構成管理ツールである。

---

# Ansibleという選択肢

[Ansible](https://www.ansible.com/) は、SSHベースで外部端末を操作できる自動化ツールである。

## 基本要件

- SSH接続できること
- Pythonがインストールされていること（多くのLinuxは標準搭載）

主な対象：

- Linux
- macOS

※ WindowsもWinRM経由で対応可能。

Linuxであれば、ほぼデフォルト要件を満たしている。

---

# なぜAnsibleがよいのか？

## 1. 対象側の開発が不要

最大の利点はここ。

操作される側にAPIを実装する必要がない。

- SSHができればOK
- エージェント不要（エージェントレス）

つまり、

> コントロール側だけ実装すればよい

外部端末を「そのまま」使える。

---

## 2. エラー処理が整理されている

単なるscp + sshスクリプトと違い、

- 成功
- 失敗
- 到達不能（unreachable）

が明確に区別される。

さらに、

- どのホストが失敗したか
- どのタスクで止まったか

が標準ログとして取得できる。

独自にログ設計をする必要がない。

---

## 3. YAMLで可読性が高い

AnsibleはYAMLで処理を書く。

```yaml
- name: 設定ファイルを配置
  copy:
    src: config.conf
    dest: /etc/app/config.conf

- name: サービス再起動
  service:
    name: app
    state: restarted
```

- 処理の流れが自然言語に近い
- コメントも書ける
- ドキュメント性が高い
- アプリコードよりも「運用設計書」に近い。

## 4. 冪等性が確保される

Ansibleの思想は **冪等性（Idempotency）** にある。

- 変更がなければ何もしない  
- 必要な変更だけを行う  

これは単なるシェルスクリプトとの大きな違いである。

例えばシェルスクリプトでは、

- 毎回ファイルを書き換える
- 毎回サービスを再起動する
- 状態を考慮しない

といった処理になりがちだ。

一方Ansibleでは、

- 設定に差分がある場合のみ更新
- 変更があった場合のみ再起動

といった「状態ベース」の制御ができる。

そのため、

> 安全に何度でも実行できる

という強い特性を持つ。

これは運用自動化において非常に重要なポイントである。

---

## 5. 複数端末のグループ管理

Ansibleのインベントリ機能により、対象ホストをグループ化できる。

例えば：

- 開発環境
- 検証環境
- 本番環境

といった単位で管理が可能だ。

さらに、

- 全台一斉変更
- 特定グループのみ変更
- 1台だけピンポイント実行

といった柔軟な操作ができる。

これは単なるSSHスクリプトでは管理が破綻しやすい部分であり、  
台数が増えるほど差が顕著に出る。

---

# 結論

APIがない外部端末を操作する場合、

- 自前でSSH自動化を書くか
- 構成管理ツールを使うか

という選択になる。

小規模・単発用途であればスクリプトでも問題ない。

しかし、

- 複数台運用
- ログ管理
- エラーハンドリング
- 再実行性
- 将来の拡張

を考えるなら、Ansibleは非常に現実的で堅実な選択肢になる。

## 内容
今回は、外部端末側にファイルを送信するコマンドを書いた。

外部端末に  
/var/www/test.txtファイルが生成されるので、実行する際は注意



##
必要なライブラリをインストール
``` bash
pip install ansible ansible-runner
```

## ディレクトリ構成
sample  
├── sample.py           : 実行スクリプト  
└── sample.yaml : play-bookファイル


### 実行スクリプト
``` python
import ansible_runner


# playbookの実行
r = ansible_runner.run(
    private_data_dir='/home/user1/workdir/ansible_app/sample',
    playbook='success_sample.yaml',
    inventory={
        "dev-servers": {
            "hosts": {
                "pc-1": {
                    "ansible_host": [試したいPCのIPアドレス],
                    "ansible_connection": "ssh",
                    "ansible_user": [PCのユーザー名],
                    "ansible_password": [PCのパスワード],
                    "ansible_become_password": [PCのパスワード]
                }
            }
        }
    } # pc-1,pc-2と書くこともできるし、別ファイルに書くこともできる。
)


# 結果の確認
print(f"Status: {r.status}")
print(f"RC: {r.rc}")

# エラーハンドリング。今回はprint文で確認するのみにした
for event in r.events:
    if event.get("event") == "runner_on_unreachable":
        print(event["event_data"]["res"])

    elif event.get("event") == "runner_on_failed":
        print(event["event_data"]["res"])

```
パスワードをべた書きしないやり方もある！

#### 成功例
``` yaml
- name: 設定ファイルを設定
  hosts:
    - dev-servers
  become: True #特権権限で実行
  tasks:
    - name: 設定ファイルをコピー
      copy:
        content: "HELLO"
        dest: "/var/www/test.txt" #送り先
        
    # - name: 値チェック。 # 失敗するタスク
    #   assert:
    #     that:
    #       - 1 == 2
    #     fail_msg: "1は2ではない"
```
失敗させたいときは「値チェック。」のコメントアウトを外して実行してみてね！
